# 产品判断记录

## 产品核心

### 用户
- **目标用户**: AI coding agent
- **使用场景**: 创建可靠的 harness 结构
- **用户痛点**: 文档与实际不符、维护成本高
- **质量门槛**: 所有文档必须可验证、可执行

### 任务
- **核心任务**: 为项目创建生产级治理框架
- **成功标准**: agent 能够独立理解和操作项目
- **验证方式**: 通过验证脚本和质量评分

## 产品规则

### 1. 可靠性优先于功能
- 功能再强大，不可靠就没用
- 宁可少功能，也要高可靠
- 验证证据比自信更重要

### 2. 歧义视为 spec 缺口
- 任何模糊的需求都是缺陷
- 需要明确的验收标准
- 不能有两种理解方式

### 3. 仓库里找不到的事实视为运行上不存在
- 所有重要信息必须在仓库中
- 外部文档必须同步到仓库
- 口头决策必须记录到文档

## 禁区模式

### 1. 隐蔽破坏性操作
- 不执行用户未明确批准的破坏性操作
- 覆盖文件前必须确认
- 删除操作必须有备份

### 2. 静默失败
- 所有错误必须明确报告
- 不能忽略错误继续执行
- 必须有错误处理机制

### 3. 无真相来源的状态
- 所有状态必须有文件记录
- 不能依赖聊天历史
- 不能依赖内存中的状态

### 4. 无法一句话解释的功能
- 每个功能必须能一句话解释清楚
- 如果解释不清楚，说明设计有问题
- 需要简化设计

## 产品原则

### 1. 用户至上
- 以用户需求为导向
- 解决用户实际问题
- 提供清晰的使用指南

### 2. 简单可靠
- 设计简单明了
- 实现可靠稳定
- 维护成本低

### 3. 可扩展
- 支持不同规模的项目
- 支持不同的技术栈
- 支持不同的团队结构

## 产品指标

### 核心指标
- **完成率**: agent 能够独立完成任务的比例
- **重试次数**: 完成任务需要的平均重试次数
- **缺陷数**: 发现的缺陷数量

### 质量指标
- **文档完整性**: 所有文档都有具体内容
- **验证通过率**: 验证命令通过的比例
- **一致性**: 文档间信息一致的程度

## 产品演进

### 短期目标（1-3个月）
- 完善五子系统模型
- 提高文档质量
- 增强验证机制

### 中期目标（3-6个月）
- 支持更多项目类型
- 提供更多模板
- 增强智能检测

### 长期目标（6-12个月）
- 成为行业标准
- 支持团队协作
- 提供可视化界面
