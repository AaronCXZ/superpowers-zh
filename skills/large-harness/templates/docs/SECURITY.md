# 安全规则

## Secrets 与凭证规则

### 1. 不硬编码任何凭证
- 不在代码中硬编码密码、API Key、Token
- 不在配置文件中硬编码敏感信息
- 不在日志中记录敏感信息

### 2. 记录加载路径
```bash
# 使用环境变量
export DATABASE_URL="postgresql://user:password@localhost/db"

# 使用 .env 文件
DATABASE_URL=postgresql://user:password@localhost/db

# 使用密钥管理服务
aws secretsmanager get-secret-value --secret-id prod/db/password
```

### 3. 使用环境变量或密钥管理服务
- **开发环境**: 使用 .env 文件
- **测试环境**: 使用环境变量
- **生产环境**: 使用密钥管理服务

## 不可信输入处理

### 1. 验证所有外部输入
```typescript
// 验证用户输入
function validateUserInput(input: string): boolean {
  // 检查输入长度
  if (input.length > 1000) {
    return false;
  }
  
  // 检查特殊字符
  if (/[<>\"'&]/.test(input)) {
    return false;
  }
  
  return true;
}
```

### 2. 永远不要信任用户输入
- 所有用户输入都是不可信的
- 必须进行验证和清理
- 必须进行转义处理

### 3. 使用参数化查询防止注入
```typescript
// 错误：字符串拼接
const query = `SELECT * FROM users WHERE id = ${userId}`;

// 正确：参数化查询
const query = 'SELECT * FROM users WHERE id = $1';
const result = await db.query(query, [userId]);
```

## 外部动作审批规则

### 1. 破坏性操作需要明确批准
- 删除数据需要用户确认
- 覆盖文件需要用户确认
- 重置配置需要用户确认

### 2. 生产环境变更需要额外审批
- 数据库变更需要 DBA 审批
- 部署变更需要负责人审批
- 配置变更需要运维审批

### 3. 记录所有外部动作
```json
{
  "action": "DELETE_USER",
  "userId": "123",
  "timestamp": "2026-07-02T16:20:00Z",
  "operator": "admin",
  "reason": "用户请求删除账户"
}
```

## 依赖与评审规则

### 1. 新依赖需在 plan 中说明理由
- 为什么需要这个依赖
- 有什么替代方案
- 对现有系统的影响

### 2. 安全改动需显式验证
- 安全相关代码需要代码审查
- 安全配置需要验证
- 安全测试需要通过

### 3. 定期更新依赖版本
- 每月检查依赖更新
- 每季度更新依赖版本
- 及时修复安全漏洞

## 数据安全

### 1. 数据分类
- **公开数据**: 可以公开访问的数据
- **内部数据**: 仅内部访问的数据
- **敏感数据**: 需要特殊保护的数据
- **机密数据**: 最高级别保护的数据

### 2. 数据保护措施
- **加密**: 敏感数据加密存储
- **访问控制**: 基于角色的访问控制
- **审计**: 记录数据访问日志
- **备份**: 定期备份数据

### 3. 数据保留策略
- **保留期限**: 根据法规要求确定
- **删除策略**: 过期数据安全删除
- **归档策略**: 历史数据归档保存

## 网络安全

### 1. HTTPS 强制
- 所有通信使用 HTTPS
- 证书定期更新
- HSTS 头配置

### 2. 防火墙配置
- 仅开放必要端口
- 限制访问来源
- 监控异常流量

### 3. 入侵检测
- 监控异常登录
- 监控异常请求
- 监控数据泄露

## 安全事件响应

### 1. 事件分类
- **P0**: 数据泄露、系统被入侵
- **P1**: 安全漏洞被利用
- **P2**: 安全配置错误
- **P3**: 安全日志异常

### 2. 响应流程
1. **发现事件**: 监控系统检测到异常
2. **确认事件**: 安全团队确认事件
3. **遏制事件**: 限制事件影响范围
4. **根除事件**: 消除事件原因
5. **恢复事件**: 恢复正常运营
6. **总结事件**: 总结经验教训

### 3. 通知机制
- **内部通知**: 通知相关团队
- **外部通知**: 通知受影响的用户
- **监管通知**: 通知相关监管机构

## 安全合规

### 1. 法规要求
- **GDPR**: 欧盟通用数据保护条例
- **CCPA**: 加州消费者隐私法案
- **HIPAA**: 健康保险流通与责任法案

### 2. 行业标准
- **ISO 27001**: 信息安全管理标准
- **SOC 2**: 服务组织控制报告
- **PCI DSS**: 支付卡行业数据安全标准

### 3. 合规检查
- 定期进行合规审计
- 记录合规状态
- 及时修复不合规项
