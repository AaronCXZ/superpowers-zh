# 前端治理

## UI 原则

### 1. 清晰优先
- 功能比视觉更重要
- 用户能理解界面在做什么
- 错误信息清晰易懂

### 2. 可发现/可重走/可重启
- 用户能随时重启流程
- 关键操作有明确的入口
- 支持断点续传

### 3. 复用组件优先
- 避免重复实现相同功能
- 使用设计系统组件
- 保持一致性

## 护栏

### 设计系统参考
- **组件命名规则**: 使用 PascalCase，如 `UserProfile`
- **间距字体 token**: 使用设计系统定义的间距和字体
- **状态变体**: 组件状态清晰定义
- **可访问性要求**: 符合 WCAG 标准

### 关键用户状态全覆盖
- **加载状态**: 显示加载指示器
- **错误状态**: 显示错误信息和重试按钮
- **空状态**: 显示引导信息
- **成功状态**: 显示成功反馈

### 一致性
- 相同组件相同行为
- 相同操作相同反馈
- 相同错误相同处理

## 验证要求

### 关键用户旅程证据
1. **截图/视频证明功能正常**
   - 核心功能有截图证明
   - 关键流程有视频记录

2. **浏览器验证**
   - 真实交互测试
   - 跨浏览器兼容性测试

3. **视觉回归标准化**
   - 确保 UI 变更符合预期
   - 自动化视觉回归测试

### 验证命令

```bash
# 运行前端测试
npm test

# 运行视觉回归测试
npm run test:visual

# 运行可访问性测试
npm run test:a11y

# 运行跨浏览器测试
npm run test:cross-browser
```

## 组件开发规范

### 文件结构
```
src/components/
├── ComponentName/
│   ├── index.tsx          # 组件主文件
│   ├── ComponentName.tsx  # 组件实现
│   ├── styles.module.css  # 样式文件
│   ├── types.ts           # 类型定义
│   └── __tests__/         # 测试文件
```

### 命名规范
- **组件文件**: PascalCase，如 `UserProfile.tsx`
- **样式文件**: camelCase，如 `styles.module.css`
- **类型文件**: camelCase，如 `types.ts`
- **测试文件**: 与组件文件同名，如 `UserProfile.test.tsx`

### 导出规范
- 默认导出组件
- 命名导出类型和工具函数
